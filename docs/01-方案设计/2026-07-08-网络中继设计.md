# NetRelayTool 设计文档（网络中继调试 + 录制回放）

- **日期**：2026-07-08
- **状态**：已确认（brainstorming 逐条确认）
- **模块**：`src/tools/NetRelayTool/`
- **归属**：DeviceForge Tool（Backend + Widget 配对）

---

## 1. 背景与目标

现场需要一个网络调试工具，在**不影响生产数据流运行**的前提下，串接在数据生产者与真实消费者之间，
原封不动地双向转发数据，同时**捕获**原始流量用于观察，并**录制**流量以便日后**回放**。

### 数据流模型（透明中继代理）

```
数据生产者(TCP/UDP客户端) ──①原始数据──▶ 工具(监听) ──②原封不动──▶ 真实消费者
数据生产者               ◀──④转发─────  工具        ◀──③使用/回控制── 真实消费者
                                         │
                                         ├──▶ 实时 Hex+ASCII 视图
                                         └──▶ 录制(.nrec 自定义格式)
                                                  │
                                                  ▼
                                            回放引擎:按原始时序重放"上行"数据给消费者
                                                     (模拟生产者,用于无生产者时驱动消费者测试)
```

### 术语（全文统一）

| 术语 | 定义 | 在代码中的角色 |
|------|------|---------------|
| 数据生产者 (Producer) | 主动连接工具的 TCP/UDP 客户端 | client（工具 accept 的一端） |
| 真实消费者 (Consumer) | 工具主动去连接的对端，消费数据/回控制 | upstream（工具 connect 的一端） |
| 上行 (Upstream dir) | 生产者 → 消费者 方向的数据 | `RelayDirection::Upstream` |
| 下行 (Downstream dir) | 消费者 → 生产者 方向的数据 | `RelayDirection::Downstream` |

> 注意：方向枚举名 `Upstream/Downstream` 沿用现有代码，语义 = "流向消费者 / 流向生产者"。

## 3. 架构

沿用 DeviceForge Tool 双层模式：`NetRelayBackend`(ServiceTask) + `NetRelayWidget`(QWidget)，
通过 `std::function` 回调 + `QMetaObject::invokeMethod(Qt::QueuedConnection)` 跨线程解耦。

新增两个内聚单元（**独立于中继引擎，可单独测试**）：

```
NetRelayBackend (中继引擎, 已有)
   │  录制:每次 dataCb 时把 {方向,相对时间戳,字节} 交给 Recorder
   ▼
RelayRecorder (新增)         RelayPlayer (新增)
   写 .nrec 文件               读 .nrec 文件 + 定时器按时序重放
   ─────────────               ──────────────────────────────
   open(path)                  load(path) → 校验 magic/版本
   append(dir,ts,bytes)        connectConsumer(host,port,proto)
   close()                     start()/pause()/stop()
                               信号:进度/状态/单条重放
```

### 3.1 职责边界

- **RelayRecorder**：只负责把中继过程中的数据块**顺序写盘**。不懂 socket，不懂 UI。输入 = (方向, 相对时间戳 ms, QByteArray)。
- **RelayPlayer**：只负责**读取 .nrec 并按时序把"上行"记录重发到消费者**。持有自己的 QTcpSocket/QUdpSocket（不复用中继的 socket，避免与实时中继冲突）。
- **NetRelayBackend**：中继引擎（已有），录制时把数据块转交 Recorder。**回放与实时中继互斥**（见 §6 状态机）。
- **NetRelayWidget**：UI。新增"录制"开关与状态、"回放"面板。

## 4. 录制文件格式（.nrec v1）

自定义小端二进制。设计目标：可精确回放、实现简单、单遍顺序写。

### 4.1 文件头（固定 32 字节）

| 偏移 | 大小 | 字段 | 值 |
|------|------|------|-----|
| 0 | 4 | magic | `"NREC"` (0x4E 0x52 0x45 0x43) |
| 4 | 2 | version | `1` |
| 6 | 1 | protocol | 0=TCP, 1=UDP |
| 7 | 1 | reserved | 0 |
| 8 | 8 | start_epoch_ms | 录制开始的绝对时间(ms)，用于展示，非回放依据 |
| 16 | 16 | reserved | 0 |

### 4.2 记录条目（变长，顺序追加）

| 大小 | 字段 | 说明 |
|------|------|------|
| 1 | direction | 0=上行(生产者→消费者), 1=下行(消费者→生产者) |
| 1 | reserved | 0 |
| 4 | session_id | 会话号（多会话录制时区分） |
| 8 | ts_offset_ms | 相对录制开始的毫秒偏移（回放时序依据） |
| 4 | length | payload 字节数 |
| length | payload | 原始字节 |

> 回放时序依据 `ts_offset_ms` 的**差值**（相邻记录间隔），保留原始节奏。

---

## 5. 回放语义

- **默认只回放"上行"记录**（direction=0）到**消费者**（模拟生产者）。下行记录在文件中保留但首期回放忽略（用于日后扩展"回放给生产者"）。
- **连接**：回放前用户输入/确认消费者地址端口（默认取录制时的消费者地址，可改）。TCP 建连后开始；UDP 直接 writeDatagram。
- **时序**：用 QTimer 按相邻 `ts_offset_ms` 差值调度下一条；首期支持"原速"，UI 预留倍速下拉（1x/2x/尽快），后端接口带 `speedFactor` 参数。
- **控制**：开始 / 暂停 / 停止；进度回调（已回放 N/总 M 条、当前时间偏移）。
- **互斥**：回放期间禁止启动实时中继，反之亦然（同一 Tool 内单活跃模式）。

## 6. 状态机（Backend 顶层）

```
        ┌──────── Idle ────────┐
        │                      │
   startRelay()           startReplay()
        │                      │
        ▼                      ▼
    Relaying                Replaying
   (可选录制中)          (原速重放上行)
        │                      │
   stopRelay()            stopReplay()
        │                      │
        └────────▶ Idle ◀──────┘
```

- Idle → Relaying：`startRelay()`（可带 `recordPath` 开启录制）
- Idle → Replaying：`startReplay(nrecPath, consumerHost, consumerPort)`
- 两态互斥；进入任一态前校验当前为 Idle，否则 reportError。

## 7. UI 设计（NetRelayWidget 扩展）

在现有布局基础上增加：

```
┌─ 中继配置 ─────────────────────────────────────────┐
│ 协议[TCP▼] 监听[127.0.0.1:9000] → 消费者[ip:port]   │
│ [▶启动] [■停止]  ☑录制到文件[...路径...]            │  ← 新增录制勾选+路径
├─ 回放 ────────────────────────────────────────────┤  ← 新增整行
│ 录制文件[...选择.nrec...] 消费者[ip:port] 速率[原速▼]│
│ [▶回放] [⏸暂停] [■停止]  进度[====----] 12/40      │
├─ 会话列表 ──────────┬─ 十六进制视图 ────────────────┤
│ (QTreeWidget)       │ (Hex+ASCII, 方向着色)          │
├─────────────────────┴───────────────────────────────┤
│ 日志                                                  │
└──────────────────────────────────────────────────────┘
```

- 录制：勾选后 startRelay 时打开 Recorder；停止时关闭并提示文件路径。
- 回放：独立分区；回放进行时禁用"启动中继"按钮，反之亦然。
- 安全：`.nrec` 含原始流量，导出/另存沿用现有二次确认 + 文件权限限制。

## 8. 与现有实现的差距（对比结论）

| 需求 | 现状 | 动作 |
|------|------|------|
| 透明中继(TCP/UDP,双向,原样) | ✅ 已实现且通过审查 | 保留 |
| 实时 Hex+ASCII 视图 | ✅ 已实现 | 保留 |
| 文本 hex 导出 | ✅ 已实现（人类可读，非回放格式） | 保留为"导出快照"功能，与录制并存 |
| **录制为可回放格式(.nrec)** | ❌ 缺失 | **新增 RelayRecorder** |
| **回放引擎(按时序重放上行给消费者)** | ❌ 缺失 | **新增 RelayPlayer + UI + 状态机** |

> 现有中继骨架约占整体 70% 且质量已验证；本设计新增的录制/回放是区别于普通抓包工具的核心增量。

## 10. 测试策略

- **RelayRecorder 单元测试**：写入若干 (方向,ts,bytes) → 读回校验字节与偏移一致。
- **RelayPlayer 单元测试**：构造 .nrec → 回放到本地 echo 消费者 → 校验收到字节序列与上行记录一致；校验相邻间隔≈原始 ts 差值（容差）。
- **端到端**：本地起 echo 服务器作消费者，nc/脚本作生产者 → 中继+录制 → 停止 → 回放 → 消费者两次收到相同数据。
- **格式健壮性**：损坏/截断的 .nrec 需被 load() 拒绝并报错，不崩溃。
