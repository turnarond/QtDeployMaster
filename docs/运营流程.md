## 阶段 1：基建期（立即做，1-3 天）

### ⚠️ 最紧急的一件事：许可证

你的 README 写的是"许可：内部使用"，但项目已经公开到 GitHub 了。**这是一个法律层面的矛盾**——没加开源许可证的代码，默认是"保留所有权利"，别人理论上不能合法使用、复制、分发你的代码，即使放在 GitHub 上。

更要注意你的第三方依赖：libcurl 是 MIT/X 衍生许可，lwlog/lwcommunicate 这些 lw* 库的许可也要逐一确认。需要做的事：

1. **选一个许可证**。工具类项目推荐 `MIT`（最宽松，传播无阻力）或 `Apache-2.0`（带专利保护，工业场景更稳）。如果你希望别人改进也必须开源，选 `GPL-3.0`
2. 在仓库根目录加 `LICENSE` 文件
3. README 里的"许可"字段改成对应的许可证
4. 如果有些 lw* 库是你公司内部的、不方便开源，那这部分代码要么不放 GitHub，要么单独说明授权方式——**不能整个项目标"内部使用"又公开源码**

### GitHub 仓库门面

README 已经写得相当专业了，但工业软件的用户做决定很看重"长什么样"。补几个东西能显著提升转化率：

- **界面截图 + 操作 GIF**：放在 README 顶部特性区下面。工业仪表盘深色主题是你的卖点，不展示太可惜。一张拖拽部署的 GIF 比 500 字描述管用
- **Quick Start 章节**：目前 README 直接跳到 CMake 构建，对只想"先跑起来看看"的用户门槛偏高。加一段"下载预编译版 → 解压 → 运行"的最短路径
- **GitHub topics**：加 `qt` `industrial-automation` `plc` `iot` `devops-tools` `scp` `remote-deployment` 这些标签，这是 GitHub 搜索和推荐的核心依据
- **Issue / PR 模板**：在 `.github/` 下加模板，引导用户报告问题时附上设备类型、协议、Qt 版本。工业场景的 bug 复现特别依赖环境信息

- **3 分钟操作演示**：用 OBS 录一段"从配置设备 → 拖拽文件 → 批量部署"的完整流程。不用精致，能看清操作就行。放 B 站 + README 里。工业用户特别吃"眼见为实"这一套

## 几个容易踩的坑

1. **别在半成品状态大力推广**。你功能表里 5 个模块还是"旧模式"，如果第一批用户冲着某个旧模式模块来，发现体验不一致，差评会劝退后来者。建议推广文案聚焦已迁移的"批量文件部署"作为主打，其他模块标注"持续迁移中"，管理预期

2. **密码不持久化这个设计**，对个人试用是安全考虑，但推广时会劝退一批"想每天用"的用户。可以考虑加一个"记住密码（本地加密存储，可选）"的选项，在文档里说明安全权衡

3. **CI 是加分项，但不是推广重点**。你的 GitHub Actions 对开发者有意义，对终端用户（运维工程师）无感。别在面向用户的文章里花篇幅讲 CI

4. **记录运营数据**。用 GitHub 的 Insights 看克隆数、star 趋势、referring sites。哪篇文章带来的流量最多，就多做那个方向。这是低成本的"增长 hacking"
